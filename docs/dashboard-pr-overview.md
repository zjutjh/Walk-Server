# Dashboard 模块 PR 宏观说明

## 1. 目的与范围
本次改动主要补齐 Dashboard 侧核心查询接口，并统一了实现范式，目标是：

- 降低接口延迟：以短 TTL Redis 缓存承接高频读请求。
- 保证统计口径一致：将聚合规则沉淀到 repo 层，API 层只负责参数校验、编排和返回。
- 提升可维护性：按 API -> Repo -> Cache 分层，减少 SQL 与业务判断散落在 Handler 的情况。

本次重点覆盖了：

- 队伍筛选与详情
- 总览统计（overview）
- 路线统计（全量与单路线）
- 点位/路段统计

## 2. 架构与分层
Dashboard 相关逻辑当前采用三层结构：

- API 层（api/dashboard/**）
  - 负责参数绑定与校验
  - 负责缓存命中短路
  - 负责结果组装与错误码返回
- Repo 层（dao/repo/dashboard/**）
  - 负责 SQL 聚合、筛选与统计口径
  - 统一沉淀跨接口复用的查询能力
- Cache 层（dao/cache/dashboard/**）
  - 负责 key 规范、TTL、读写封装
  - 与 API 层解耦，避免散落的 Redis 访问逻辑

## 3. 关键数据流（宏观）
以大多数 Dashboard 接口为例，统一遵循如下流程：

1. 参数校验：校验必填字段、枚举值、分页参数合法性。
2. 读缓存：命中后直接返回，避免数据库聚合压力。
3. 回源查询：缓存未命中或异常时，走 repo 聚合查询。
4. 组装响应：将数据库结果映射到 API Response 结构。
5. 回写缓存：短 TTL 回填，覆盖接下来一段时间的热点请求。

该流程在统计类接口中尤为关键，可在高并发下显著减少重复聚合查询。

## 4. 统计口径与业务约定
为保证前后端理解一致，当前实现采用以下宏观约定：

- 统计对象以 people 为主（人数口径），并通过 team 关联路线。
- 路线有效性由 routes 表控制（is_active + campus）。
- 路段筛选依赖 route_edges（route_name + prev_point_name + point_name）。
- 走错人数与状态人数分开统计，避免口径污染。
- 排序策略优先保证审计可读性和输出稳定性（如 time + id）。

## 5. 缓存策略（短 TTL）
Dashboard 查询以读多写少、可容忍轻微短时延迟为主，因此使用短 TTL：

- 统计总览/单路线/筛选等接口均采用秒级缓存。
- 读缓存失败时不会中断请求，降级回源数据库。
- 缓存解析失败会自动忽略并回源，避免脏缓存导致功能不可用。
- 与队伍状态变更相关的接口在必要场景会主动删除相关缓存，降低短时不一致窗口。

## 6. 可审阅性与扩展性
从 PR 评审角度，建议关注以下点：

- 逻辑边界是否清晰：API 编排、Repo 聚合、Cache 封装是否分离。
- 新增查询是否可复用：是否避免重复 SQL 与重复口径定义。
- 口径变化是否可追踪：状态映射、走错统计、人数口径是否集中定义。
- 失败降级是否安全：缓存失败是否仅告警、数据库错误是否正确返回错误码。

## 7. 已知限制与后续建议
当前实现优先保证可用性和一致风格，后续可考虑：

- 将状态映射口径进一步收敛到公共函数，减少多处维护。
- 为关键聚合查询补充 explain 与索引审查，持续优化慢查询风险。
- 针对高峰流量接口补充基准测试与缓存命中率观测。
- 增加面向 dashboard 的集成测试样例（参数校验、分页游标、缓存回源路径）。
